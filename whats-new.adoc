=== Reprise après sinistre



==== La prise en charge de Google Cloud VMware Engine (GCVE) est désormais généralement disponible (GA)

NetApp Disaster Recovery pour GCVE est désormais disponible en disponibilité générale (GA). Avec la prise en charge GA, Disaster Recovery pour GCVE prend désormais en charge :

* *Instantanés gérés par la plateforme :* Gérez les sauvegardes et les politiques de rétention de manière transparente via ONTAP ou Google Cloud Platform (GCP).
* *Plusieurs banques de données par plan de réplication :* Simplifiez vos plans de reprise après sinistre en consolidant plusieurs banques de données dans une seule stratégie de réplication.
* *Forcer le basculement en cas d'indisponibilité du site source :* Basculer vers un site de reprise après sinistre si la source est indisponible.
* *Différer la resynchronisation SnapMirror et restaurer la réplication à la demande :* Différez les resynchronisations SnapMirror pour éviter tout impact sur les charges de travail de production pendant les périodes de pointe grâce à l’option « skip protect », puis restaurez la réplication à la demande ultérieurement avec l’option « re-protect ».


Pour plus d'informations sur la prise en charge de GCVE, découvrez link:https://docs.netapp.com/us-en/data-services-disaster-recovery/release-notes/dr-limitations.html["Configurations prises en charge pour NetApp Disaster Recovery"^].

==== Restauration propre

Ransomware Resilience prend désormais en charge les restaurations propres, une nouvelle option de récupération qui vous aide à vous remettre rapidement et facilement des attaques de ransomware. Les restaurations propres offrent des flux de travail guidés qui identifient les points de restauration optimaux pour remettre vos charges de travail en service tout en minimisant l'interruption. Les flux de travail dans la restauration propre fournissent également des informations au niveau fichier sur vos données, garantissant que vous pouvez restaurer des charges de travail exemptes de logiciels malveillants, ce qui empêche toute réinfection et minimise la perte de données.

Pour en savoir plus sur les configurations prises en charge et sur la manière de configurer votre système pour une restauration complète, consultez link:https://docs.netapp.com/us-en/data-services-ransomware-resilience/task-clean-restore.html["Créer un environnement de récupération isolé"^]. === Cloud Volumes ONTAP



==== Possibilité de gérer les paramètres proxy de l'agent NetApp Console pour Cloud Volumes ONTAP

Vous pouvez désormais gérer les paramètres de proxy sur l’agent NetApp Console pour Cloud Volumes ONTAP, même si vous perdez la connexion ou si le proxy est mal configuré. Auparavant, si l’agent Console ne pouvait pas se connecter à Cloud Volumes ONTAP pendant 20 minutes, il remplaçait vos paramètres de proxy manuels par les paramètres par défaut. Cela provoquait des échecs de communication, y compris des problèmes avec les messages AutoSupport. Pour conserver vos paramètres de proxy pour les systèmes existants, exécutez l’appel d’API suivant :

`PUT /occm/config`

Incluez le paramètre suivant dans le corps de la requête :

[source, json]
----
{
  "proxyMode": "No_Overwrites"
}
----
Le mode par défaut est standard, ce qui signifie que l'agent de la Console remplace vos paramètres de proxy par les valeurs par défaut s'il ne parvient pas à se connecter à Cloud Volumes ONTAP pendant 20 minutes.

https://docs.netapp.com/us-en/console-automation/cvo/put-occm-config.html["Configurer les paramètres modifiables de la NetApp Console"^]

=== Classification des données



==== Version 1.52

.Validation des ressources
Pour garantir la fiabilité, Data Classification vérifie désormais la mémoire et les ressources d'espace disque disponibles lors de l'analyse, de la génération de rapports et des actions. Lorsque l'utilisation de la mémoire dépasse 95 % ou que l'espace disque est inférieur à 6 Go, Data Classification interrompt automatiquement les opérations. Les opérations reprennent dès que les ressources sont suffisantes.

Pour surveiller les ressources de votre système, consultez link:https://docs.netapp.com/us-en/data-services-data-classification/task-health-monitor.html["Surveiller l'état de Data Classification"^].

.Limite accrue pour la copie et le déplacement
Les fonctions de copie et de déplacement sont désormais prises en charge pour les destinations CIFS. La limite de taille par fichier pour ces deux actions a été portée à 1 Go contre la limite précédente de 50 Mo. Les limites d'opération pour ces deux actions ont été portées à 1 To ou 5 millions de fichiers.

Pour plus d'informations, voir link:https://docs.netapp.com/us-en/data-services-data-classification/task-copy-move.html["Copier, déplacer et supprimer des données"^].

.Autorisations de configuration d'intégration Active Directory
Les modifications apportées aux paramètres d'intégration d'Active Directory (AD) dans Data Classification sont désormais réservées aux administrateurs. Les spectateurs n'ont plus la possibilité de modifier les paramètres AD.

Pour plus d'informations sur le rôle de visionneur de classification, voir link:https://docs.netapp.com/us-en/console-setup-admin/reference-iam-predefined-roles.html#data-service-roles["Rôles du service de données"^].

.Prise en charge de RHEL 9.7
Data Classification a ajouté Red Hat Enterprise Linux v9.7 pour l'installation manuelle sur site, y compris les déploiements sur des sites isolés.

Les systèmes d'exploitation suivants nécessitent l'utilisation du moteur de conteneurs Podman, et ils nécessitent Data Classification version 1.30 ou supérieure : Red Hat Enterprise Linux version 8.8, 8.10, 9.0, 9.1, 9.2, 9.3, 9.4, 9.5, 9.6 et 9.7.

.Prise en charge du proxy transparent
La classification des données prend désormais en charge les configurations de proxy transparent, permettant un déploiement plus fluide dans les environnements réseau proxy ou restreints.

Pour plus d'informations sur le déploiement dans des environnements restreints, voir link:https://docs.netapp.com/us-en/data-services-data-classification/task-deploy-compliance-dark-site.html["Installer NetApp Data Classification sur un hôte Linux sans accès à Internet"^].

=== Cloud Volumes ONTAP



==== Prise en charge de Cloud Volumes ONTAP pour les machines virtuelles Google Cloud de nouvelle génération

Avec la version 9.18.1, NetApp fait passer les nouveaux déploiements Cloud Volumes ONTAP des machines virtuelles N2 à la nouvelle génération de machines virtuelles Google Cloud série C3 pour une expérience plus rapide et plus évolutive. Vous pouvez désormais tirer parti des machines virtuelles série C3 lors du déploiement de Cloud Volumes ONTAP 9.18.1 et versions ultérieures sur Google Cloud. Les machines de la série C3 offrent des performances améliorées et des limites de capacité plus élevées grâce à l'utilisation de Google Virtual NIC (gVNIC) et de disques Hyperdisk Balanced, garantissant des performances dynamiques pour les charges de travail intensives.


NOTE: Pour le moment, Cloud Volumes ONTAP prend en charge la série C3 uniquement dans les déploiements à nœud unique.

Si votre système Cloud Volumes ONTAP exécute la version 9.18.1 ou ultérieure, les packages préconfigurés que vous utilisez pour des déploiements sur un seul nœud utilisent automatiquement des machines virtuelles C3, tout en vous permettant de personnaliser les paramètres d'IOPS et de débit en fonction des besoins de votre charge de travail. De même, lors de la création d'agrégats, vous pouvez ajouter des disques Hyperdisk Balanced pour obtenir de meilleures performances et une meilleure évolutivité dans Google Cloud. De plus, vous pouvez sélectionner les variantes LSSD des machines de la série C3 pour une prise en charge par défaut du Flash Cache.

Il est impossible de modifier le type de disque des machines virtuelles C3 lors de l'ajout de volumes aux agrégats, car C3 ne prend en charge que les disques Hyperdisk Balanced. De même, lors de la réplication d'un système avec des machines virtuelles N2 vers une machine virtuelle C3, le type de disque est défini par défaut sur Hyperdisk Balanced.

==== Sécurité VNet pour Cloud Volumes ONTAP dans Azure

Les déploiements Cloud Volumes ONTAP 9.18.1 et ultérieurs dans les zones de disponibilité uniques et multiples d'Azure prennent en charge le chiffrement Azure Virtual Network (VNet) dans le cadre de leur stratégie de sécurité multicouche afin de protéger les données en transit. Cloud Volumes ONTAP exploite le protocole Datagram Transport Layer Security (DTLS) natif d'Azure pour sécuriser la communication entre les nœuds ONTAP, les interfaces de gestion et les autres services Azure, empêchant l'interception et l'accès non autorisé. Ce chiffrement au niveau du réseau complète les protections intégrées d'ONTAP pour le stockage et les données au repos afin d'assurer la sécurité de bout en bout de vos données.

https://review.docs.netapp.com/us-en/storage-management-cloud-volumes-ontap_gcp-sn-c3-support/reference-networking-azure.html#networking-for-azure-vnet-encryption["Mise en réseau pour le chiffrement Azure VNet"^]

La classification des données prend désormais en charge la création de catégories personnalisées pour vos données. Pour plus d'informations, voir link:https://docs.netapp.com/us-en/data-services-data-classification/task-custom-classification.html["Créer une classification personnalisée"^].

.Définition de données obsolètes personnalisée
La classification des données vous permet désormais de personnaliser la définition des données obsolètes afin qu'elle corresponde aux besoins de votre organisation. Auparavant, les données obsolètes étaient définies comme toutes les données dont la dernière modification remontait à trois ans. Désormais, les données obsolètes peuvent être identifiées en fonction de leur dernière consultation ou de leur dernière modification ; cette période peut aller de 6 mois à 10 ans.

Pour plus d'informations, voir link:https://docs.netapp.com/us-en/data-services-data-classification/task-stale-data.html["Personnaliser la définition des données obsolètes"^].

.Performances améliorées
Les temps de chargement de toutes les pages de la section Classification des données, du rapport de cartographie des données et des filtres de la page Investigation ont été réduits.

.Délai estimé pour les rapports d'enquête
Lorsque vous téléchargez un rapport d'enquête, la classification des données affiche désormais le temps estimé pour la fin du téléchargement.
